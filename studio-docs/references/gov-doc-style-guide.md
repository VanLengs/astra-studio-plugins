# 政府信息化项目文档写作规范参考

> 本文档为 studio-docs 写作技能的参考资料，按需加载。涵盖政府公文格式、信息化项目文档标准、常用表达、图表规范等。

---

## 一、政府公文写作规范

### 1.1 标题层级体系

政府信息化项目文档采用以下标题层级，与 Markdown 标题级别对应：

| 层级 | 格式 | 字体 | 字号 | Markdown |
|------|------|------|------|----------|
| 文档标题 | 居中、加粗 | 方正小标宋简体 | 二号 | `# 标题` |
| 一级标题 | 顶格、加粗 | 黑体 | 三号 | `## 一、标题` |
| 二级标题 | 顶格、加粗 | 楷体_GB2312 | 三号 | `### （一）标题` |
| 三级标题 | 缩进2字符 | 仿宋_GB2312 | 三号 | `#### 1. 标题` |
| 四级标题 | 缩进2字符 | 仿宋_GB2312 | 三号 | `##### （1）标题` |

### 1.2 编号体系

严格遵循以下编号递进规则：

```
一、二、三、……                  （一级）
（一）（二）（三）……            （二级）
1. 2. 3. ……                     （三级）
（1）（2）（3）……               （四级）
①②③……                         （五级，仅在表格或注释中使用）
```

**注意事项：**
- 同一层级编号必须连续，不得跳号
- 一级编号后用顿号"、"，不用点号
- 三级编号后用下脚点"."
- 括号编号为全角括号，括号外不加标点

### 1.3 正文格式要求

- **字体**: 仿宋_GB2312 或仿宋
- **字号**: 三号（16pt）正式公文；小四（12pt）附件和技术文档
- **行距**: 固定值 28-30 磅（正式公文）；1.5 倍行距（技术文档）
- **段前段后**: 0 行
- **首行缩进**: 2 个汉字字符
- **页边距**: 上 3.7cm、下 3.5cm、左 2.8cm、右 2.6cm（标准公文）
- **技术文档页边距**: 上下 2.54cm、左右 3.17cm

## 二、信息化项目文档标准体系

### 2.1 适用国家标准

| 标准编号 | 标准名称 | 适用场景 |
|----------|----------|----------|
| GB/T 8567-2006 | 计算机软件文档编制规范 | 软件项目全套文档 |
| GB/T 9385-2008 | 计算机软件需求规格说明规范 | 需求分析报告 |
| GB/T 16680-2015 | 系统与软件工程 文档管理指南 | 文档管理流程 |
| GB/T 22032-2008 | 系统与软件工程 系统生存周期过程 | 项目全生命周期 |
| GB/T 25000.51-2016 | 软件产品质量要求与评价 | 验收测试报告 |
| GB/T 36073-2018 | 数据管理能力成熟度评估模型 | 数据治理方案 |
| GB/T 35273-2020 | 信息安全技术 个人信息安全规范 | 安全设计方案 |

### 2.2 项目文档全套清单

一个完整的政府信息化项目通常包括以下文档体系：

```
项目前期：
├── 可行性研究报告
├── 项目建议书
├── 需求分析报告
└── 项目概算/预算

项目实施：
├── 建设方案（总体设计）
├── 详细设计文档
├── 实施方案
├── 项目管理计划
└── 安全保障方案

项目验收：
├── 测试报告
├── 用户手册
├── 运维手册
├── 培训方案
└── 验收报告

招投标：
├── 招标文件
├── 投标文件（技术标 + 商务标）
└── 评标报告
```

### 2.3 文档间引用关系

文档间必须保持数据一致性和逻辑连贯性：

- **可行性研究报告** → 为**建设方案**提供建设必要性论证和投资估算依据
- **需求分析报告** → 为**建设方案**提供功能需求清单和非功能需求指标
- **建设方案** → 为**投标文件**提供技术路线和架构设计参考
- **建设方案** → 为**详细设计**提供总体架构约束
- **项目概算** → 必须与**可行性研究报告**中的投资估算一致
- **测试报告** → 测试用例必须覆盖**需求分析报告**中的全部功能点
- **验收报告** → 验收标准必须追溯到**建设方案**中的建设目标

**一致性检查要点：**
- 项目名称、建设单位在所有文档中完全一致
- 建设内容清单在各文档中范围一致（可详略不同）
- 投资金额在概算、可研、招标文件中逻辑一致
- 建设周期在各文档中口径一致
- 技术指标在需求、方案、测试报告中可追溯


## 四、政策引用格式规范

### 4.1 标准引用格式

```
《政策文件全称》（发文字号）
```

**示例：**
- 《国务院关于加快推进"互联网+政务服务"工作的指导意见》（国发〔2016〕55号）
- 《"十四五"国家信息化规划》（中网办发文〔2021〕26号）
- 《数字中国建设整体布局规划》（中共中央 国务院，2023年2月）

### 4.2 引用规则

1. **首次引用**使用全称 + 发文字号，之后可用简称并加注"（以下简称《XX》）"
2. **书名号**使用中文书名号《》，不使用引号
3. **发文字号**格式：发文机关代字〔年份〕序号，使用六角括号〔〕
4. **多部门联合发文**按发文顺序列出全部机关名称
5. **引用密度建议**：
   - 项目背景章节：3-5 项核心政策
   - 建设必要性章节：2-3 项直接相关政策
   - 技术方案章节：1-2 项技术标准或规范
   - 保障措施章节：1-2 项管理制度依据

### 4.3 政策时效性

- 优先引用最近 3 年内的政策文件
- 国家标准引用最新版本，标注标准编号和年份
- 地方性政策需注明省/市级别
- 已废止或替代的政策不得引用

## 五、数据表格规范

### 5.1 表格格式要求

| 要素 | 规范 |
|------|------|
| 表题 | 居中，表序 + 表名，如"表3-1 系统功能模块清单" |
| 表头 | 加粗，居中对齐，底色可用浅灰 |
| 字体 | 表内文字使用仿宋或宋体，小四号（12pt） |
| 边框 | 全边框（实线，0.5pt），表头下方可用 1pt 粗线 |
| 对齐 | 文字左对齐，数字右对齐，表头居中 |
| 单位 | 在表头或表注中统一标注单位 |

### 5.2 数据精度要求

- **金额**: 保留 2 位小数，单位为"万元"（大额用"亿元"）
- **百分比**: 保留 1 位小数，如 98.5%
- **时间**: 具体到月，如"2024年6月"
- **数量**: 整数不保留小数
- **性能指标**: 标注测试条件和基准

### 5.3 表格编号规则

- 采用"章节号-表序号"格式，如"表3-1""表4-2"
- 表格编号在全文中连续（同一章内连续）
- 每个表格必须有表题
- 表格在正文中必须有引用，如"详见表3-1"

## 六、Mermaid 图表在正式文档中的使用规范

### 6.1 适用场景

Mermaid 图表适合用于以下场景，在 markdown 源文件中编写，导出时渲染为图片：

| 图表类型 | 适用内容 | Mermaid 语法 |
|----------|----------|-------------|
| 流程图 | 业务流程、审批流程 | `flowchart TD` |
| 时序图 | 系统交互、接口调用 | `sequenceDiagram` |
| 甘特图 | 项目进度计划 | `gantt` |
| 架构图 | 系统架构、网络拓扑 | `flowchart TD` + subgraph |
| 类图 | 数据模型、实体关系 | `classDiagram` |
| 状态图 | 状态流转 | `stateDiagram-v2` |

### 6.2 图表格式要求

- **图题**: 图表下方居中标注，格式"图X-Y 图名"，如"图3-1 系统总体架构图"
- **图注**: 如有需要，在图题下方添加简要说明
- **编号**: 采用"章节号-图序号"格式，全章连续
- **引用**: 正文中必须引用，如"系统总体架构如图3-1所示"
- **尺寸**: 导出图片宽度不超过页面正文宽度（约 14.5cm）
- **清晰度**: 导出 PNG 分辨率不低于 200 DPI

### 6.3 Mermaid 编写规范

```markdown
<!-- 图表标记格式 -->
<!-- figure: 图3-1 系统总体架构图 -->

` ` `mermaid
flowchart TD
    subgraph 用户层
        A[PC端] --> C[统一门户]
        B[移动端] --> C
    end
    subgraph 应用层
        C --> D[业务应用系统]
    end
    subgraph 数据层
        D --> E[数据中心]
    end
` ` `
```

**注意：**
- 节点文字使用中文，简洁明了
- subgraph 名称对应架构分层
- 连线标注接口或数据流名称
- 避免过多节点（单图不超过 20 个节点）
- 复杂架构拆分为多张图

## 七、文档一致性检查清单

### 7.1 全文一致性

- [ ] 项目名称在文档标题、正文、页眉中完全一致
- [ ] 建设单位名称全称统一，简称首次出现时加注
- [ ] 建设周期（起止时间）在所有章节中一致
- [ ] 投资金额合计数与分项之和一致
- [ ] 技术指标在需求章节和方案章节中对应

### 7.2 跨文档一致性

- [ ] 项目名称在所有文档中一致
- [ ] 建设内容范围在各文档中口径一致
- [ ] 投资估算/概算/预算之间逻辑一致
- [ ] 建设目标可从可研追溯到验收标准
- [ ] 技术路线在建设方案和投标文件中一致

### 7.3 格式一致性

- [ ] 标题层级编号连续，无跳号
- [ ] 图表编号连续，无遗漏
- [ ] 所有图表在正文中有引用
- [ ] 引用政策格式统一（全称 + 发文字号）
- [ ] 表格样式统一（边框、字体、对齐）
- [ ] 页眉页脚内容正确

## 八、模板段落示例

### 8.1 项目背景段落模板

> 随着{领域}数字化转型的深入推进，{建设单位}深入贯彻落实《{核心政策}》（{发文字号}）精神，紧紧围绕{上级要求/规划目标}，以{关键理念}为导向，统筹推进{建设方向}建设。当前，{现状描述}，{问题描述}，亟需通过{解决方向}，实现{建设目标}，为{服务对象}提供{价值描述}。

### 8.2 技术方案段落模板

> 本项目采用{技术路线}，基于{技术架构}进行总体设计。系统整体架构分为{N}层：{逐层说明}。其中，{核心层}采用{关键技术}，支持{核心能力}，满足{性能指标}的要求。通过{技术手段}实现{业务目标}，有效解决{具体问题}，预计可{量化效益}。

### 8.3 保障措施段落模板

> 为确保项目顺利实施，从{保障维度1}、{保障维度2}、{保障维度3}等方面建立全方位保障体系。在{维度1}方面，{具体措施}；在{维度2}方面，{具体措施}；在{维度3}方面，{具体措施}。通过{总结手段}，切实保障项目建设目标如期实现。
